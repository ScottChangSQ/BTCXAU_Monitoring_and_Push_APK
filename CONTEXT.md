## 近期关键决定和原因
- 这轮继续细化时，不再新增另一套方向性方案，而是把现有总方案拆成 8 个里程碑和方法级改造清单：原因是总方向已经定清，如果继续停留在原则层，后续执行时仍会在“先改哪一个类、先删哪一个旧分支、测试先补到哪里”为单点反复停顿；把里程碑、状态流转、方法改造点和测试矩阵一次拆清，才能真正进入可执行状态。
- “统一真值”和“首帧本地优先”不能拆成两轮各做一半，必须在同一份总方案里一起落：原因是只做统一真值，页面仍会先空后填；只做首帧本地优先，旧的多真值和多刷新链又会把本地显示内容很快纠正掉，最终仍然会出现体感抖动和口径不一致，所以这两件事必须并成一条实现主线。
- “正确统一法”本轮正式收口为“统一底稿和 revision，而不是让所有周期成品常驻热跑”：原因是用户当前要的是图表、账户页、悬浮窗统一口径且同步感一致，但如果把所有周期都做成实时热更新，会显著放大 CPU、流量和维护成本；因此最小完整方案应当是统一市场底稿、统一账户真值、统一产品级运行态，并让页面只按当前所需周期和当前相关产品做派生与刷新。
- 这轮直接删除 `Notion Data Desk` 主题和设置中的“主题设置”入口，而不是继续修补主题细节：原因是用户已明确该主题完全未达到预期，继续保留入口只会留下无效配置和错误预期，最短正确路径就是收口回单一默认主题。
- 这轮不靠猜测源码继续试，而是统一以 ADB 截图、UI dump 和真机结果为准：原因是用户指出的是“真实页面仍有大量黑底和白字残留”，只有真机证据才能区分是 palette 本身错误，还是页面首帧时机、旧 drawable、旧 adapter 或按钮 tint 残留。
- 账户页首帧深色的修复点放在 `initializePageContent()` 之后立即补刷 `applyPaletteStyles()`，而不是继续只等后续刷新：原因是新建账户页时 `pageController` 绑定晚于宿主页显示时机，首帧如果不主动补刷，就会先落回 XML 默认深色。
- 账户页主卡片改为直接设置 `CardView` 自身底色和圆角，而不是继续对卡片调用 `setBackground(...)`：原因是 `CardView` 的真实面板颜色不受普通背景替换控制，继续改外层 background 只会留下看起来没变的深色卡片。
- 共享按钮样式里明确清除 `backgroundTint`，而不是只替换浅色 drawable：原因是之前很多按钮并不是背景资源没换，而是新 drawable 又被 tint 回黑色，必须在共享样式层一起收口。
- 这轮先收口 `账户 / 分析` 主路径和异常列表弹窗，不继续同时改桥接深页或全部历史 adapter：原因是用户当前明确指出的是 `账户 / 分析 TAB` 图表和 `Notion Data Desk` 主主题不匹配，最小完整方案应先把真实主链页面和会直接看到的异常弹窗统一到设计稿，再决定是否继续清理旧兼容路径。
- 新主题实现这轮先采用“保留金融专业风 + 新增 Notion Data Desk”两套主题，而不是直接把全部主题系统删成单主题；原因是用户只明确要求删除 4 个无效主题并新增新的 UI 主题，因此当前最小完整实现是保留仍有效的金融专业风，同时把旧 4 套无效主题从设置入口和运行时主题 id 映射里一起清掉。
- `Notion Data Desk` 方向已经正式定稿并写入设计稿，而不是继续停留在网页草图阶段；原因是用户已确认这版方向可用，后续真实开发需要有稳定的正式文档作为唯一视觉依据。

- 本轮分支整理时，正式提交只纳入业务代码、测试、设计文档和项目文档，不纳入 `.codex/config.toml`、`.superpowers/.../state/events` 这类本机环境或过程记录：原因是这些内容不影响产品功能，却会把个人工作台状态带进仓库，破坏 `main` 的可复用性。
- BMW 深色版第二轮修正规定“按键背景不用渐变，同级按钮和同级文字必须严格统一”：原因是用户明确要求工业精密感要来自秩序和一致性，而不是来自渐变或局部特殊样式；继续混用不同按钮高度、不同字重和不同文字色会让界面重新回到拼装感。
- 这轮视觉重做改为采用 `BMW` 深色 showroom 版，而不是此前讨论过的 `Mistral` 暖色浅底方案：原因是用户明确要求继续保留深色体系，只借 BMW 的工业精密感、直角几何、白灰文字层级和蓝色交互高亮语言，不把主界面切成浅底。
- 真机验证阶段出现的 `app died` 不能只看关键词下结论，必须结合 `force-stop` 操作时间和 `ApplicationExitInfo` 原因一起判断：原因是这类系统清理记录和真实 Java 崩溃都会出现在 `logcat`，如果不对时间线和退出原因，容易把“用户主动停止进程”误判成应用闪退。
- 服务端账户发布态不再继续维护 `v2_sync_state + v2_bus_state` 两套并行状态，而是收口到单一 `account_publish_state`：原因是原来一份负责运行态序号、一份负责 bus 序号，职责重叠且容易在清缓存、成交后发布、测试兼容时互相打架。
- 成交或会话切换后的“清缓存”和“清发布序号”必须分开：原因是如果把 `busSeq` 一起清零，客户端手上的 `lastBusSeq` 会失去连续性，真实变更可能被误看成 heartbeat。
- 账户持仓页展示工厂不再保留 `AccountRuntimePayload / AccountRuntimeSnapshotStore`：原因是这层只做机械拷贝，不产生新真值，继续保留只会让账户页在正式 cache 之外再长出一套重复页面运行态。
- 这轮新的实施计划把“服务器账户运行态收口 -> 客户端账户链删重 -> 统一产品运行态唯一化 -> 图表/UI 绑定频次下压”固定成执行顺序：原因是如果不先收服务器和客户端真值边界，后面继续优化图表和悬浮窗，只会更快地消费一份仍然重复组装的数据。
- 这轮先只做“数据传输 / 更新架构现状审视”，不直接改代码：原因是用户当前要先判断客户端与服务器主链里还有哪些重复拉取、重复组装、缓存层过多或职责边界不清的问题，再决定下一轮怎么收口。
- 图表页 dialog zone 现在也纳入 `ChartRefreshScheduler` 单独合帧，而不是继续同步立即刷新：原因是交易快捷条和全局状态按钮/弹层属于自己的 UI 区块，不应继续跟着主图或状态源的每次变化直接重绑。
- 统一运行态的产品展示名现在优先接受账户快照里已有的产品名，短名再按产品代码回退到 `BTC/XAU`：原因是账户条目、悬浮窗和图表摘要需要同一份产品展示口径，但展示名可能是中文，短名识别又必须继续可靠。
- 这轮新的逐文件计划只继续处理桥接页残留、统一运行态删重、图表弹层解绑和最后真机验证，不再重复第一轮 UI/性能收口：原因是当前真正没闭环的工作已经收缩成少量尾线，继续混做会放大范围并增加回归风险。
- 不保留独立仪表盘页：原因是用户已明确不需要首页行情概览，也不保留默认品种快捷卡，继续保留仪表盘会让首页失去明确职责。
- 主壳常驻入口固定为 `交易 / 账户 / 分析`，`设置` 改成独立入口：原因是设置不是高频切换场景，继续占用常驻底部导航只会挤压主交易链路。
- 异常提醒整合进图表页，而不是独立占位：原因是异常与价格、时间、仓位决策本来就强关联，放回图表主场更符合用户判断路径。
- 状态弹层继续复用现有会话、连接和异常数据源：原因是先收口真实状态链路，避免为了新 UI 再复制一套并行状态模型。
- 账户页聚合层不再占据首屏主位置：原因是用户确认账户页首屏应优先展示当前账户对象摘要和历史入口，而不是再增加一层产品聚合阅读成本。
- 账户页历史成交入口改为底部抽屉而不是分析页跳转：原因是这块需求属于账户页自己的完整历史查看，不需要强制切换一级导航，同时产品、方向、时间排序筛选都可以在轻层级里完成。
- 分析页核心统计采用“摘要 + 展开明细”而不是直接恢复整页长统计：原因是用户需要更多分析数据，但仍希望分析首页保持摘要式结构，不重新退回旧版长页面。
- 分析页“查看完整分析”入口必须显式绕过桥接回主壳逻辑：原因是摘要页到深页属于同一分析域内的下钻，如果继续套用旧桥接规则，用户会被送回当前 tab，看起来像按钮失效。
- `AccountStatsBridgeActivity` 既然继续复用 `ContentAccountStatsBinding.bind(binding.getRoot())`，就必须在旧深页布局里持续满足共享绑定类的必需视图集合：原因是共享布局新增摘要页专属控件后，桥接页如果缺少同名视图会在启动阶段直接崩溃。
- 分析页的曲线卡片属于首屏模块，不能继续跟着“次屏延后挂载”策略一起晚出现：原因是信息架构调整后它已经是一级阅读内容，延后显示会造成页面结构抖动和感知卡顿。
- 分析页首屏改成摘要页，完整历史统计继续复用旧 `AccountStatsBridgeActivity` 深页：原因是这样能在不复制数据链路的前提下，完成“一级页更克制、重内容下沉”的目标。
- `MarketChartActivity` 只保留桥接职责，不再继续维护第二套真实图表实现：原因是应用内部主链已经落在 `MainHostActivity + MarketChartFragment + MarketChartScreen`，继续保留重实现只会制造双轨维护。
- 图表叠加层签名不能再依赖 `cache.updatedAt`：原因是 `updatedAt` 每次运行态刷新都会变化，会把“内容没变、时间戳变了”的轻量更新误判成整套叠加层需要重算。
- 历史成交锚点必须改成基于 K 线窗口的二分定位：原因是 1 分钟图和大窗口场景下，线性扫描会把历史成交加载速度拖成和条数成正比。
- 账户运行态应用链不能再做“写库后立刻读回再建缓存”：原因是这会把原本内存即可完成的轻量刷新变成高频磁盘 I/O 回环，直接拖慢图表、持仓和悬浮窗同步。
- `ChainLatencyTracer` 默认关闭，远程会话 debug 统一改成 no-op：原因是这些日志已经不再参与当前主链判断，继续高频输出只会放大主线程压力和噪音。
- 服务端市场 K 线缓存寿命必须略长于 stream 推送节奏：原因是 `v2/stream` 默认 1 秒推一次，如果 `/v2/market/candles` 也按 1 秒或更短缓存，就会在主界面与悬浮窗并发读取时反复打上游。
- 第二阶段优化当前先采用“统一产品运行态 + 图表局部刷新预算”这条最短路径：原因是用户已明确后续追极限速度优先继续压图表页 UI 绑定频次，而不是再叠新缓存层。
- 统一运行态除了 `positionCount / pendingCount / totalLots / netPnl` 外，还必须给 UI 直接提供方向手数和展示名称：原因是悬浮窗、图表摘要、账户条目如果各自再判断方向和简称，就会重新出现口径分叉。
- 图表页高频刷新除了实时尾部外，叠加层和摘要也要同帧合并，但不能再新增缓存层：原因是当前真正的浪费在重复 UI 绑定，而不是底层真值缺缓存。
- `AccountStatsBridgeActivity` 这轮先改成“真实运行路径委托给 `AccountStatsScreen`，旧字段和旧字符串继续保留”而不是一步删净：原因是当前仓库仍有大量桥接页源码测试锁定老字段和老方法名，先切真实宿主再删兼容残留，风险更低。
- 账户持仓/挂单条目接统一运行态时，只允许在“同产品只有 1 条当前条目”时复用运行态口径：原因是统一运行态目前是产品级聚合结果，若直接覆盖多笔同产品单项，会把产品聚合值误显示成每一笔的单笔值。
- 重构后的回归基线改成“验证新架构职责边界”而不是“维持旧 Activity 细节”：原因是这轮失败主要来自历史测试仍在锁定旧五页结构和旧主页面职责，继续保留会误报回归并阻塞后续合并。
- 交易页状态按钮未展开时不显示异常信息：原因是异常已经在图表标记和全局状态弹层里展示，顶部入口继续堆第三段文案会压缩顶部栏可读性。
- 设置页不再保留“Tab 页管理”：原因是用户已经确认设置不再是一级导航的一部分，继续保留旧 tab 名称和显隐开关会让当前信息架构自相矛盾。
- 设置入口不能只改主壳不改旧独立页：原因是当前 App 里仍有若干桥接页和旧深页会展示自己的底部导航，只改主壳会让手机上继续看到旧 `设置` tab。
- 账户总览不再展示累计指标，当前持仓摘要也不再展示开仓价格：原因是账户首屏需要先表达当前状态与当前仓位结果，累计口径和开仓价属于更深层信息，不应继续占用首屏摘要密度。
- 按钮圆角和文字裁切要一起在共享样式层处理：原因是这两类问题都来自默认按钮样式与固定高度的组合，只改单个布局会反复漏掉同类按钮。
- 按钮和选项的统一不能只锁高度，还要把宽度分配改成按文案测量结果自适应：原因是中文文案长度差异很大，继续依赖等宽或手写权重会反复出现某一项被挤压、另一项空出来的情况。
- 弹框是否真的方角不能只靠主题 overlay，要把弹框根卡片和列表项卡片自己的圆角/描边也显式锁死：原因是列表项这类独立 `MaterialCardView` 可能绕开窗口层收口，看起来仍像旧样式残留。
- 这类“模块/弹窗/抽屉统一边框”需求必须收口到全局 surface 规则，而不是继续在单个布局或控制器里补：原因是边框颜色、宽度和背景归属一旦分散到多处，后续再调视觉尺度时必然出现遗漏和不一致。
- 下拉菜单文字重叠要按“是否存在外层覆盖标签”分成两类公共入口处理：原因是带覆盖标签的 Spinner 需要隐藏自身折叠态文字，而普通 Spinner 仍要直接显示当前选项，不能为了修一类问题把全部下拉都改成空白。
- 连续控件的横向间距不能继续散落在各页面的 `4dp/6dp/8dp` 局部 margin 里：原因是这类视觉节奏后续会反复调整，必须抽成统一 token，并把分段按钮组这类运行时布局也纳入同一规则。
- 字号层级和通用文字颜色不能继续一半放在样式里、一半散落在 XML/Java 的临时值里：原因是这类视觉尺度后续一定会继续调整，只有把主路径文本等级和选中/未选中文字色一起抽到共享规则层，后面改动才不会反复漏页。
- 交易页顶部 `产品 / 交易 / 挂单 / 状态` 四个入口必须视为同一组控件统一刷色，而不是继续让 Spinner、模式按钮和状态按钮分别维护自己的背景与文字色：原因是它们在信息架构上属于同一排一级操作，继续分散处理会反复出现“高度相同但观感不属于一组”的割裂感。
- 后续每次完成改动后默认执行“编译调试包并安装到当前真机”：原因是这能把源码层验证和实际设备验证收成同一个闭环，避免停在“只编译未上机”的中间状态。
- 当前持仓和挂单折叠条目的首行必须使用统一的左右两列结构，右侧操作按钮列宽度固定、左侧摘要列吃剩余宽度：原因是这两类条目共用同一套列表视觉，继续让按钮列按内容自适应会造成不同条目之间的操作区左右漂移，看起来不对齐。
- 共享条目摘要文字如果目标是单行展示，必须同时满足“摘要 TextView 显式锁单行尾省略”和“右侧操作列不预占未使用宽度”：原因是只做其中一边都会出现看似右侧还有空间、左侧却先换行的假性拥挤。
- 挂单价格去小数只作用于挂单折叠态摘要，不影响展开态、交易弹窗和图表标注：原因是用户这轮目标是提升列表扫描效率，而不是改变交易相关精度表达，不能把摘要层的简化误扩散到执行和分析链路。
- 这类横向操作按钮如果目标是“贴近屏幕左右边界但保留 5dp 外边距”，应优先使用 `0dp + weight` 分配整行宽度，而不是继续保留 `wrap_content`：原因是 `wrap_content` 只能收缩内容本身，无法让按钮组自然占满可用横向空间。
- 前后台切换时如果需要重建 OkHttp transport，只能同步替换客户端引用，旧连接池与调度线程释放必须放到后台线程：原因是 `evictAll()/shutdown()` 在部分机型上会触发真实 socket close，主线程执行会被 StrictMode 直接判成 `NetworkOnMainThreadException`，而 App 图标点击和悬浮窗点击都会走这条前台恢复链路。
- 全局视觉规范从“小直角 + 窄边距”切到“中圆角 + 平衡间距”：原因是用户这轮目标不再只是修文字裁切，而是按 `linear.app` 的秩序把整套界面收口为长期可复用的统一尺度，继续保留 2dp 直角会让顶部 4 控件和各模块始终像临时拼接。
- 交易页图表右上角统一只保留持仓监控摘要，不再并存 `ms` 延迟或独立摘要卡：原因是用户要的是交易决策直接相关的持仓结果，延迟文案和重复摘要会占掉图表主场的右上角信息位。
- 交易页快捷下单不再复用旧弹窗作为主入口，而是新增图表主场内的快捷模式条：原因是用户明确要一键执行和拖线挂单，旧弹窗只适合保留给复杂操作。
- `交易` 与 `挂单` 必须是互斥模式，挂单类型根据价格线相对当前价自动判断：原因是这样模式边界清晰，用户不需要在图表主场再手选 `limit / stop`，也不会同时看到两套快捷入口。
- 挂单价格线由 `KlineChartView` 直接承担绘制和拖动，不单独再造一层浮动控件：原因是价格线必须跟随图表价格坐标和视口一起变化，放在图表内部才能保持价格映射和交互一致。
- 悬浮窗标题把“手数”提到产品名同一行，并与盈亏分成上下两行分别着色：原因是方向信息和结果信息不是同一维度，拆开后阅读更快，也避免第一行只看见产品名、第二行又混合手数与盈亏。
- 悬浮窗产品名称继续收口成 `BTC/XAU` 而不是 `BTCUSD/XAUUSD`：原因是悬浮窗空间有限，用户在这里看的是产品识别和仓位结果，不需要重复看到交易侧的完整代码后缀。
- 标题下的说明文字直接删除而不是保留隐藏控件：原因是这轮目标就是收紧一级页面信息密度，继续保留说明控件只会增加共享绑定复杂度和后续误恢复风险。
- 历史成交锚点在 1 分钟周期要按分钟桶开头定位，而不是继续保留秒级时间：原因是 1 分钟已经是当前图表的最细时间槽，继续保留秒级时间会让历史成交点落在槽位内的细分位置并出现不可见风险；更高周期则仍保留真实时间，兼顾可见性与时间表达。
- 全局状态弹窗里的三颗操作按钮要在布局层直接锁定单行和统一内边距：原因是这三个入口属于同一操作组，单靠等权重不足以避免中文文案在窄设备上换行，必须把文本行数和按钮内容区约束一起固定。
- 全局状态弹窗里的三颗操作按钮不能只靠 `maxLines` 限制：原因是 `wrap_content` 高度和 MaterialButton 的运行时测量仍可能把第三个按钮撑高，所以必须同时锁定固定高度和运行时单行约束，才能真正消掉不对齐。
- 全局状态弹窗里的三颗操作按钮除了单行约束，还要有自动缩字：原因是用户要求文案必须完整显示，单行后如果空间仍紧张，就应该优先缩字保全文案，而不是继续省略。
- 全局状态弹窗里的三颗操作按钮在中文文案不等长时，不能强求完全等权重：原因是“异常列表”明显长于“设置/日志”，继续等宽会浪费前两项空间并挤压第三项，所以最终要把第三项分到更多宽度来保全文案。
- 异常交易提醒与服务常驻通知必须分开处理：原因是用户要保留异常发生时的即时弹出提示，但不要状态栏长期挂着“服务运行中”，所以即时提醒链路继续保留，服务常驻通知单独收口。
- 固定高度输入框不能直接复用按钮那套“去字体内边距 + 强行上下 padding”写法：原因是按钮文案和输入框文本的基线计算不同，继续混用会把输入文字挤偏并出现裁切。
- 这轮不继续沿用此前那套“平衡型中圆角”视觉：原因是用户已明确把目标改成更硬朗的终端式界面，要求所有按钮、模块、tab 和弹窗统一为方角、黑底或近黑深灰底、无边框，并把模块边距进一步压到 2dp，所以必须连运行时主题刷色一起收口，而不能只改 XML token。
- 仅改共享主题 token 仍不足以彻底消除圆角：原因是项目里还有不少旧 `drawable` 和 Material 弹窗窗口壳会绕过共享 token 直接生效，所以这轮必须把资源背景文件与 `AlertDialog / BottomSheetDialog` 外层容器也一起统一成方角。
- 分析页首屏不再保留两个只负责跳转到同一完整分析页的模块：原因是这两块内容对用户来说是重复入口，但它们后面的收益表和统计图区仍然有价值，所以应当保留完整图表与表格，只移除重复入口本身。
- 交易统计卡里的字段型统计不应和核心统计展开区重复并存：原因是同一批指标在同一屏重复出现只会增加阅读噪音，交易统计卡更适合只承接切换后的图表与分布结果。
- 设置页不再保留“连接与诊断”独立入口和详情区块：原因是连接状态、日志与诊断信息已经在其它位置可达，继续保留只会制造重复阅读路径。
