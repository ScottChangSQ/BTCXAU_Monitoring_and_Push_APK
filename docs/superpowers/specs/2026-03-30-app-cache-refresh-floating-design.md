# App 刷新、历史缓存与悬浮窗优化设计

## 1. 背景

当前 App 已经完成韩国服务器接入、MT5 网关接入，以及“后台摘要接口 + 前台全量接口”的第一轮流量优化。但图表刷新、历史数据保留、设置页清缓存、悬浮窗展示范围、监控页默认行为这几块仍然存在使用体验问题：

- 图表页在进入页面、切换周期、刷新过程中，触发点不统一。
- 历史 K 线刷新前会出现空白，且本地缓存仍有上限，不符合“拉过就一直保留”的要求。
- 历史交易数据没有明确独立的长期缓存与清理入口。
- 当前持仓顶部总盈亏样式不够醒目。
- 悬浮窗仍以 BTC/XAU 行情为主，没有展示所有 MT5 持仓品种的盈亏。
- 行情监控页还保留“是否开始监控”开关，不符合“默认监控”的目标。

## 2. 本次目标

本次实现要同时满足以下 8 项需求：

1. 启动 App、切换到 K 线图表页、切换时间周期时，均刷新图表。
2. 图表最右侧 K 线与边框距离增加到图表区域的 `1/7`。
3. 当前持仓界面最上面的总盈亏按红绿显示，并调整文字大小。
4. 刷新图表时保留历史数据，不再先清空后显示。
5. 历史数据拉取后立即存储，后续只增量补充，不自动删除。
6. 设置页“清除缓存”新增“清除历史交易数据”“清除历史行情数据”选项。
7. 悬浮窗显示 MT5 当前所有持仓品种的分产品盈亏。
8. 行情监控界面删除“是否开始监控”按钮，默认监控；悬浮窗同步删除“是否监控”信息，仅保留连接状态。

## 3. 已确认边界

- 历史行情缓存按“交易对 + 时间周期”分别保存。
- 只要拉取过的历史行情，就一直保留，不设置自动上限。
- 历史交易数据也应长期保留，后续只做增量补充。
- 悬浮窗中的分产品盈亏范围是 MT5 当前实际所有持仓品种，而不是只看 BTC/XAU。

## 4. 方案对比

### 方案 A：在现有结构上做增量保留（推荐）

保留当前页面结构、网关调用关系和主要类分工，只调整缓存策略、刷新触发点和悬浮窗展示模型。

优点：

- 改动集中，能尽量复用当前代码。
- 风险最低，适合本轮直接落地。
- 不影响已经跑通的服务器与网关链路。

缺点：

- 本地历史文件会持续增长，占用手机存储空间。

### 方案 B：抽象统一本地仓库

把行情历史、历史交易、账户快照、悬浮窗数据都抽成统一仓库层，然后各页面只消费统一仓库。

优点：

- 结构更整齐，后面扩展更方便。

缺点：

- 需要改动多个调用入口，风险和工期都更高。

### 方案 C：直接重构为数据库化持久层

把历史行情、历史交易、持仓快照全部迁移到数据库层再重做读取逻辑。

优点：

- 长期演进空间最大。

缺点：

- 本轮明显过重，不适合当前节奏。

## 5. 采用方案

采用方案 A：在现有结构上做“增量保留 + 统一刷新入口 + 悬浮窗扩展”。

原因：

- 当前主要诉求是先把使用体验、流量、后台可用性一起稳住。
- 现有代码已经具备 K 线缓存、账户缓存、悬浮窗和前台服务基础，只需补齐关键缺口。
- 可以在不改服务器部署方式的前提下完成。

## 6. 设计说明

### 6.1 图表刷新策略

`MarketChartActivity` 的刷新入口统一为三类：

1. App 启动后首次进入图表页。
2. 从其他页面切换回图表页。
3. 切换图表时间周期。

统一行为：

- 先立即加载本地已缓存的该交易对、该周期历史数据并展示。
- 页面保持已有数据，不先清空列表。
- 在后台发起增量补齐，只请求缺失区间和最新一段数据。
- 增量合并后重新绘制图表。

这样可以避免“刷新前先没数据”的空白感，同时减少重复下载。

### 6.2 历史行情缓存策略

`KlineCacheStore` 从“有限数量裁剪”改为“长期保留、按 key 保存、按时间去重、增量合并”。

关键规则：

- key 仍按 `symbol + interval` 区分。
- 新拉取的数据写入前先和本地历史按 `openTime` 去重合并。
- 不再自动截断旧数据。
- 页面刷新不再调用任何清空历史数据的流程。

设置页新增独立入口后，只有用户手动清理历史行情数据时才删除这些文件。

### 6.3 历史交易缓存策略

账户页把历史交易数据视为长期资产，而不是一次性展示数据。

关键规则：

- 每次从网关获取交易数据后，立即写入本地历史交易缓存。
- 后续刷新时先显示本地历史，再与服务端返回结果做增量合并。
- 合并时以稳定主键去重，例如 `dealTicket / orderId / positionId / closeTime` 组合。
- 只有用户在设置页手动点击“清除历史交易数据”时才删除。

### 6.4 图表右侧留白

`KlineChartView` 当前右侧留白逻辑调整为固定使用图表绘图区宽度的 `1/7`。

要求：

- 该留白基于价格绘图区宽度计算。
- 切周期、切品种、刷新后行为保持一致。
- “回到最新”按钮与倒计时文案的位置，也要跟着新的绘图区边界重新计算，避免覆盖到最后一根 K 线。

### 6.5 当前持仓顶部总盈亏样式

`AccountStatsBridgeActivity` 顶部总盈亏摘要继续保留现有信息结构，但增强视觉层级：

- 盈利显示绿色，亏损显示红色，持平显示主文字颜色。
- 总盈亏数字字号大于其余摘要文字。
- 分产品与库存费等其他信息保留较小字号。

目标是让用户一眼先看到核心盈亏结果，再看附属说明。

### 6.6 设置页清缓存拆分

设置页“清除缓存”弹窗拆成更明确的清理项，至少包含：

- 清除历史交易数据
- 清除历史行情数据
- 清除异常记录与预加载等临时缓存

交互上采用多选或逐项选择，避免再出现“一键全清导致历史都没了”的情况。

### 6.7 悬浮窗改造

悬浮窗的目标从“行情观察窗”升级为“连接状态 + 持仓盈亏摘要窗”。

调整内容：

- 删除“是否监控”状态文案，只保留连接状态。
- 保留已有连接状态展示能力。
- 新增 MT5 当前所有持仓品种的分产品盈亏列表。
- 分产品按当前持仓聚合，至少展示产品名和该产品总盈亏。

若悬浮窗空间不足，优先保证：

1. 连接状态
2. 当前有持仓的产品盈亏

BTC/XAU 行情展示不再作为唯一核心信息。

### 6.8 行情监控页默认监控

`MainActivity` 删除“是否开始监控”按钮后，监控逻辑改为默认启用。

行为定义：

- 服务启动即默认进入监控状态。
- 页面不再提供手动开/关入口。
- 悬浮窗不再显示“运行中/已停止”这种监控状态文案。

注意：这里删除的是“用户可见的手动开关”，不是删除后台监控能力本身。

## 7. 模块改动范围

预计涉及以下模块：

- `app/src/main/java/com/binance/monitor/ui/chart/MarketChartActivity.java`
- `app/src/main/java/com/binance/monitor/ui/chart/KlineChartView.java`

- `app/src/main/java/com/binance/monitor/ui/settings/SettingsActivity.java`

## 8. 数据流设计

### 图表页

1. 页面进入或切周期
2. 读取本地历史行情缓存
3. 立即绘图
4. 后台请求最新缺口数据
5. 合并去重
6. 写回本地
7. 更新图表

### 账户页

1. 页面请求网关快照
2. 读取本地历史交易缓存
3. 将服务端交易与本地交易做合并
4. 更新界面
5. 写回本地历史交易缓存

### 悬浮窗

1. 服务层拿到最新连接状态与账户持仓快照
2. 先聚合成“产品 -> 总盈亏”
3. 传给悬浮窗管理器
4. 悬浮窗按列表渲染

## 9. 错误处理

- 本地缓存读取失败时，不中断页面显示，按“无本地缓存”处理。
- 增量刷新失败时，保留当前已显示历史，不清空页面。
- 悬浮窗拿不到账户数据时，仅显示连接状态和“暂无持仓数据”。
- 清理缓存操作必须带确认提示，并明确说明会删掉哪类数据。

## 10. 测试策略

至少覆盖以下验证：

1. 启动 App 后进入图表页，会先显示本地历史，再补最新数据。
2. 切换图表周期时，不会先清空图表。
3. 多次刷新同一交易对同一周期，不会重复保存同一根 K 线。
4. 历史 K 线缓存不会被自动截断。
5. 历史交易缓存可以增量合并且不重复。
6. 设置页可以分别清除历史交易、历史行情。
7. 悬浮窗在存在多个 MT5 持仓品种时，可以逐项显示盈亏。
8. 行情监控页不再显示监控开关，默认监控仍有效。

## 11. 非目标

本次不包含以下内容：

- 不改韩国服务器部署结构。
- 不改 MT5 网关协议。
- 不把整个缓存体系整体迁移到数据库。
- 不额外新增新的后台推送协议。

## 12. 实施顺序

推荐按以下顺序执行：

1. 先改缓存层：历史行情长期保留、历史交易独立缓存。
2. 再改图表页：统一刷新入口、先本地后增量、右侧 `1/7` 留白。
3. 再改账户页和设置页：总盈亏样式、缓存清理选项。
4. 最后改悬浮窗和监控页：接入所有持仓品种、删除监控开关显示。
