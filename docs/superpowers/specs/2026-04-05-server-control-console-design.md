# 服务器唯一运维控制台设计

## 1. 背景

当前服务器端虽然已经具备主网关、轻量管理面板、部署脚本和基础状态接口，但实际使用仍然过度依赖命令行：

- 服务器部署后还要手动执行多个 PowerShell 命令
- 需要人工判断网关、MT5、EA、代理到底是谁有问题
- APP 连不上时，用户需要自己试地址、看端口、看日志
- 部署文件分散时，容易因为路径不一致或漏传文件导致启动失败

用户本次目标不是继续补命令，而是把服务器端的“启动、停止、重启、配置、状态、诊断、日志、建议地址”全部收进一个统一的 Web UI 中。第一次人工初始化后，后续日常不再直接接触命令行。

## 2. 设计目标

按优先级排序：

1. 一个浏览器入口完成日常运维
2. 不再要求用户记住任何命令
3. 出问题时先给中文结论，再给修复动作
4. 部署文件统一收口到单一根目录，避免路径问题
5. 尽量复用现有后台能力，缩短落地时间

## 3. 非目标

以下内容不在第一阶段范围内：

- 远程精确操控 MT5 图表窗口进行 EA 手工挂载
- 多用户权限系统
- 复杂审计系统
- 完全替换 MT5 原生 GUI
- 把所有第一次初始化动作都强行放到 UI 内完成

## 4. 总体方案

本次采用“新做一个更完整的 Web 控制台，但底层复用现有网关与管理能力”的方案。

整体结构分为两层：

### 4.1 展示层

新增一个正式的服务器控制台前端，作为唯一运维入口。

用户只面对：

- 状态卡片
- 中文结论
- 启停按钮
- 配置表单
- 诊断结果
- 日志列表

命令行不再直接暴露给用户。

### 4.2 执行层

后台继续复用并扩展当前这些能力：

- `admin_panel.py` 的状态聚合能力
- `admin_panel.py` 的组件启停能力
- `server_v2.py` 的健康检查与数据接口
- 现有计划任务和 PowerShell 脚本
- `.env` 读写与运行时缓存清理

也就是说，新控制台主要重构的是“交互层和状态表达方式”，不是重写整套后台。

## 5. 单文件夹部署要求

这是本次新增的硬性要求。

服务器部署文件必须统一收口到单一根目录，例如：

```text
C:\mt5_bundle
├─ mt5_gateway
├─ windows
└─ console
```

要求如下：

1. 所有运行所需文件都在该根目录内闭合
2. 不依赖仓库其他散落目录
3. 所有默认路径由根目录推导，不要求用户自己拼路径
4. 控制台显示的路径、脚本调用的路径、计划任务使用的路径必须一致
5. 上传时允许“整包覆盖上传”，避免漏传单文件

第一阶段交付时，新的控制台也必须包含在这个单一部署包中。

## 6. 页面结构

控制台分为 5 个主页面。

### 6.1 总览

首页一屏展示：

- 服务器状态
- 主网关状态
- MT5 客户端状态
- EA 心跳状态
- APP 连通性状态
- 最近错误
- 当前建议地址

首页只保留最常用按钮：

- 一键刷新状态
- 保存并应用配置
- 打开诊断中心

### 6.2 组件控制

对以下对象提供启动、停止、重启按钮：

- MT5 网关
- MT5 客户端
- 反向代理（Caddy / Nginx）
- 自动启动任务

每个组件卡片同时展示：

- 当前状态
- 关键说明
- 最近检查时间
- 当前是否建议重启

### 6.3 配置中心

主界面不再以原始 `.env` 文本为主，而改为中文字段表单。

主要配置分组：

- MT5 账户
- MT5 路径
- 网关主机与端口
- 公网入口与建议 APP 地址
- EA 推送参数
- 高级配置

保存后必须明确提示：

- 哪些项已变更
- 哪些组件需要重启
- 是否可以“一键保存并重启相关组件”

原始 `.env` 文本保留为高级模式入口，不作为默认编辑方式。

### 6.4 诊断中心

诊断中心的核心目标是把技术状态翻译成中文结论。

例如不直接显示“health failed / timeout”，而显示：

- 主网关未运行
- 本机网关正常，但公网入口未通过
- APP 当前填写地址错误
- EA 超过阈值未推送
- MT5 已打开但未登录目标账户

同时提供：

- 当前结论
- 建议动作
- 建议 APP 地址
- 一键重新诊断
- 跳转到相关配置页

### 6.5 日志与历史

提供以下内容：

- 最近日志
- 最近错误
- 最近操作记录
- 缓存状态
- 异常规则状态

并提供这些操作：

- 一键清缓存
- 重新拉取状态
- 导出最近日志

## 7. 首次初始化向导

用户已接受“第一次人工初始化”，但初始化后的系统仍需要一个 UI 向导来确认是否真正完成。

首次初始化向导不是替代所有手工步骤，而是做“初始化完成确认”和“剩余缺口提示”。

步骤建议如下：

1. 检查 Python 与依赖是否已就绪
2. 检查 MT5 路径与登录信息是否有效
3. 检查网关基础配置是否完整
4. 检查自动启动任务是否已注册
5. 检查本机连通性是否通过
6. 检查公网入口是否通过
7. 生成最终建议 APP 地址

向导完成后进入正式控制台，后续日常不再依赖命令行。

## 8. EA 处理策略

EA 不再设计成“让用户在控制台里远程拖拽挂到 MT5 图表”。

第一阶段采用更稳定的状态模型：

- EA 文件是否存在
- EA 最近心跳是否新鲜
- EA 推送是否正常
- 最近一次推送时间
- 当前数据源是否来自 EA

若异常，则 UI 里显示：

- EA 未就绪
- EA 已就绪但无心跳
- EA 心跳过期
- 当前已回退到其他数据源

这样可以把“EA 手工挂载”的不稳定交互，转化成“EA 是否可用”的可诊断状态。

## 9. 数据与状态模型

控制台后端需要统一聚合以下状态：

- 服务器在线状态
- 网关健康状态
- 网关来源状态
- MT5 连接状态
- EA 快照新鲜度
- 代理状态
- 计划任务状态
- 当前配置摘要
- 当前建议 APP 地址
- 最近错误与日志摘要

控制台前端一律消费统一状态接口，不让每个页面自己拼多个来源。

## 10. 动作模型

所有按钮动作都必须返回结构化结果，而不是只回原始命令输出。

统一返回内容至少包括：

- `ok`
- `summary`
- `details`
- `affectedComponents`
- `needRefresh`
- `needRestart`

例如：

- “已保存配置，需要重启：主网关、代理”
- “主网关已重启成功”
- “MT5 启动失败：路径不存在”

## 11. 错误处理原则

默认先显示中文结论，再允许展开技术细节。

错误展示优先级：

1. 当前结论
2. 建议动作
3. 影响范围
4. 技术细节（可展开）

禁止把原始 PowerShell 输出直接作为主界面文案。

## 12. 第一阶段实现范围

第一阶段必须完成：

- 新控制台首页
- 组件控制页
- 配置中心表单化
- 诊断中心
- 日志与历史页
- 首次初始化向导
- 单文件夹部署包整合
- 后端统一状态接口
- 后端统一动作接口
- 中文诊断与建议地址输出

第一阶段可以暂缓：

- 真正自动修复所有错误
- 复杂权限管理
- 多实例管理
- 远程 MT5 图表级控制

## 13. 验收标准

用户可视角度的验收标准如下：

1. 日常操作时不需要打开命令行
2. 服务器、网关、MT5、EA、APP 连通状态都能在 UI 中直接看到
3. 出现“APP 连不上”时，控制台能直接给出中文结论和建议地址
4. 保存配置后，控制台会明确提示哪些组件要重启
5. 所有部署文件统一位于单一根目录，整包上传后不会再因路径散落导致失败
