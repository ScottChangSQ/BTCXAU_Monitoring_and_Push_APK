# MT5 账户登录轻量确认收口设计方案

## 1. 背景

这次要修的不是“服务器能不能把 MT5 登上去”，而是“手机端把什么叫登录成功定义错了”。

当前系统里，登录链路已经分成两段：

1. 服务端会话段  
   `/v2/session/login` 负责受理账号密码、执行 MT5 登录探针、建立当前远程会话。
2. 客户端同步段  
   手机端在登录受理后进入“正在同步账户”，再等待账户数据回到本地页面。

问题在于，当前客户端把“登录完成”绑在了 `/v2/account/full` 这条最重的全量快照链上。

这会带来一个结构性后果：

- 服务器端 MT5 已经登录成功
- 行情 stream 正常
- 轻量账户快照正常
- 但只要 `/v2/account/full` 没有在客户端等待窗口内完成，手机端就继续显示“正在同步账户”，最终表现为 `timeout`

这和用户这次提供的现象一致：

- 第三、四阶段之前可以正常登录
- 第三、四阶段完成后开始经常 `timeout`
- 行情推送不受影响
- 部署日志里的 `/health`、`/v2/account/snapshot`、`/v2/account/history`、`wss /v2/stream` 全部健康

因此，这次设计不再继续围绕“再优化 MT5 登录探针”或“继续拉长超时”展开，而是直接修正“登录完成合同”。

## 2. 本次方案目标

本次方案只解决 4 件事：

- 把手机端“登录成功”的收口条件从 `/v2/account/full` 改为轻量账户运行态确认
- 保证“服务器端 MT5 已登录”后，手机端能尽快进入已登录/在线状态
- 保留 `/v2/account/full` 作为历史、统计、图表的补全链路，但不再让它决定登录是否成功
- 让部署检查能够更早识别“轻链路健康但 full 偏慢”的情况

## 3. 非目标

- 不新增新的 MT5 登录算法
- 不重写 `/v2/session/login` 服务端契约
- 不新增一条全新的登录确认接口
- 不用“延长客户端 timeout”代替根修
- 不为了临时兼容保留“双重登录成功定义”

## 4. 已确认事实

### 4.1 客户端当前真实行为

当前客户端主链里：

- 登录受理后会进入“正在同步账户数据”
- 前台刷新统一走 `fetchFullForUi()`，也就是 `/v2/account/full`
- 只有当 full 返回 `connected=true`，且账号、服务器与目标账户完全匹配时，才调用 `onSnapshotApplied(...)` 把会话标记为 active

也就是说，当前“登录成功”实际上等价于：

> full 全量账户快照成功返回，并完成身份匹配

### 4.2 服务端当前真实行为

当前服务端已经具备较轻的账户运行态能力：

- `/v2/account/snapshot` 可返回当前账户轻量状态
- `wss /v2/stream` 会推送 `accountRuntime`
- 客户端本地已经有 `UnifiedRuntimeSnapshotStore`
- `AccountStatsPreloadManager.applyPublishedAccountRuntime(...)` 已能消费服务端发布的运行态

也就是说，系统里已经存在一个更轻、更快的“当前账户是否在线、是谁”的真值链。

### 4.3 部署日志提供的关键信息

用户提供的最新部署日志证明：

- `2026-04-20 15:57:08` `/v2/account/snapshot` 已 `200`
- `2026-04-20 15:57:08` `/v2/account/history?range=all` 已 `200`
- `2026-04-20 15:57:09` `wss://tradeapp.ltd/v2/stream` 已收到 `syncBootstrap`

但日志没有验证 `/v2/account/full`。

因此，当前最合理的解释不是“服务器没登录上”，而是：

> 轻量链已经可用，但客户端仍然坚持等待 full 全量链完成，导致登录界面迟迟不收口

## 5. 设计原则

### 5.1 登录成功必须绑定会话真值，而不是补全真值

登录动作本质上是在建立“当前远程账户会话”。

所以登录成功应该回答的是：

- 当前有没有在线账户
- 当前在线的是不是目标账号

而不是回答：

- 历史数据是不是都补齐了
- 统计图表是不是都渲染完了

### 5.2 轻链路负责收口，重链路负责补全

这次设计固定采用下面的职责分工：

- 轻链路：决定登录是否成功
- 重链路：补历史、统计、图表、分析

两者不能再混用。

### 5.3 不新增第二套真值

这次不增加新接口，也不新建第三套“登录确认缓存”。

正式真值继续只用现有两类：

- 账户运行态 `accountRuntime / snapshot`
- 账户全量快照 `full`

变化只发生在“谁负责收口”。

## 6. 方案总览

本次采用“轻量运行态收口，full 后台补全”的单方案。

完整流程调整为：

1. 手机端提交 `/v2/session/login`
2. 服务端受理并执行 MT5 登录探针
3. 客户端进入“正在同步账户”
4. 客户端优先等待轻量账户真值到达  
   来源可以是：
   - `wss /v2/stream` 推送的 `accountRuntime`
   - 或前台主动拉取 `/v2/account/snapshot`
5. 只要轻量真值满足以下条件，就立刻判定登录成功：
   - `connected=true`
   - `account/login` 与目标账号一致
   - `server` 与目标服务器一致
6. 登录成功后，再继续后台拉 `/v2/account/full`
7. full 拉回后只更新历史、统计、图表，不再反过来决定登录是否成功

## 7. 详细设计

### 7.1 登录成功新合同

登录成功的正式定义改为：

> 当前远程账户运行态已确认在线，且其 canonical 账号身份与用户本次请求的目标账号一致

这里的“账号身份一致”包括两项：

- login/account 一致
- server 一致

如果只有 `server` 一致而 `login` 不一致，仍然不能收口。

如果 `login` 一致但 `server` 缺失，则：

- 只有在当前等待状态里本身没有显式 server 时，才允许以 login 为主收口
- 只要用户输入了 server，就必须匹配 server

这条规则继续保持当前系统已有的“身份严格匹配”原则，不降低正确性。

### 7.2 轻量确认来源

这次不新建新来源，直接复用现有来源，优先顺序如下：

1. `wss /v2/stream` 推送的 `accountRuntime`
2. `/v2/account/snapshot` 主动拉取结果

两者共享同一套身份匹配规则和同一套收口逻辑。

换句话说，不管轻量真值来自 stream 还是 snapshot，只要命中目标账号，就调用同一处“登录完成”入口。

### 7.3 `/v2/account/full` 的新职责

`/v2/account/full` 在这次调整后继续保留，但职责变为：

- 拉完整仓位、挂单、成交、曲线、统计
- 生成页面所需的强一致完整缓存
- 在交易后做较重的完整对齐

它不再承担以下职责：

- 决定登录成功
- 决定全局账户是否在线
- 作为首次登录唯一收口条件

这样做以后，即使 full 偏慢，用户也能先完成登录，再逐步看到历史和统计补齐。

### 7.4 失败语义

登录阶段的失败语义改为只围绕轻量真值判断：

- 如果服务端明确返回登录失败：立即失败
- 如果轻量链返回 `connected=false` 且给出明确错误：登录失败
- 如果轻量链暂时还没到，但网关正常：保持“正在同步账户”
- 如果轻量链已确认目标账号在线：立即成功

`full` 阶段若失败，不再映射成“登录失败”，而映射成：

- “账户已登录，完整数据同步较慢”
- 页面局部保持 loading 或显示最近可用数据

也就是说：

- 登录失败和完整数据补全失败，正式拆成两个语义

### 7.5 前台页面状态变化

登录相关页面状态改为四段：

1. `SUBMITTING`  
   正在提交登录
2. `AWAITING_RUNTIME_CONFIRMATION`  
   已受理，正在等待账户上线确认
3. `ACTIVE`  
   已确认目标账户在线
4. `FULL_SYNCING`  
   已登录，但完整数据仍在补齐

其中第 4 段是补全态，不允许再把会话打回未登录。

### 7.6 页面展示要求

这次页面文案也要跟真实状态对齐：

- 轻量确认前：`正在同步账户`
- 轻量确认后：显示已登录/账户在线
- full 补齐阶段：如果页面仍缺图表或历史，只在局部显示“正在加载完整数据”

不再使用一个总文案把“登录未完成”和“完整数据未补齐”混在一起。

### 7.7 服务端改动边界

本次服务端只允许做两类小改动：

- 若现有 `snapshot` / `accountRuntime` 某些 identity 字段不稳定，补齐或标准化字段
- 在部署检查里新增 `/v2/account/full` 的健康诊断

本次不做：

- 新增 `/v2/session/confirm`
- 重写 session manager
- 增加第二套登录状态缓存

## 8. 数据流

### 8.1 登录成功主链

1. 用户输入账号密码并提交
2. 客户端调用 `/v2/session/login`
3. 服务端完成 MT5 登录探针并受理会话
4. 客户端进入等待轻量确认态
5. stream 或 snapshot 返回目标账号运行态
6. 客户端完成身份匹配
7. 会话转为 active
8. 后台继续拉 full，补齐页面完整数据

### 8.2 登录后 full 补全链

1. 会话已 active
2. 页面或预加载器请求 `/v2/account/full`
3. 完整快照写入本地缓存
4. 历史、统计、图表刷新
5. 若失败，仅影响完整视图，不回退会话

## 9. 边界与保护

### 9.1 切换账号

切换已保存账号时，必须复用同一套轻量确认规则。

不能出现：

- 新账号 full 还没回来，就被旧账号缓存误收口

因此等待态必须继续带上目标 `login/server`，所有轻量确认都必须按目标身份做比对。

### 9.2 旧 stream 事件污染

如果登录/切换期间收到的是旧账号的 runtime 事件，必须忽略。

只有目标账号命中时，才能触发收口。

### 9.3 full 较慢

如果 full 明显偏慢：

- 登录仍然成功
- 账户在线状态仍然成立
- 页面只在依赖 full 的局部区域保留 loading

### 9.4 网关短暂离线

如果 session 已受理但网关短暂离线：

- 仍维持等待轻量确认态
- 不直接把它判成登录失败
- 等待恢复或明确错误

## 10. 验证方案

本次验证分 4 组。

### 10.1 客户端合同测试

要新增或更新测试，锁定以下规则：

- 登录成功不再依赖 `/v2/account/full`
- stream runtime 命中目标账号即可收口成功
- snapshot 命中目标账号也可收口成功
- old account runtime 不能误收口新账号
- full 失败不会把已登录状态打回失败

### 10.2 服务端合同测试

要锁定以下边界：

- `snapshot` / `accountRuntime` 在已登录时必须稳定给出 canonical `login/server`
- `/v2/account/full` 慢或失败时，不影响轻量链 identity

### 10.3 部署检查

部署脚本新增一条 `/v2/account/full` 检查，但分类为：

- `full 诊断`

它的作用是发现 full 是否异常，不再把它和登录主链混成一个判断。

### 10.4 人工验收

至少验证这几类场景：

- 新账号登录成功
- 已保存账号切换成功
- full 故意变慢时，登录仍能成功
- 行情正常、账户在线、历史稍后补齐

## 11. 风险与权衡

### 11.1 为什么不继续用 full 收口

因为 full 是最重链路。

把最重链路当登录闸门，会让登录体验被历史、统计、曲线、图表全部绑死。

这不是严格，而是职责混淆。

### 11.2 为什么不新增新接口

因为现有系统已经有 `accountRuntime` 和 `snapshot`，再新开一条确认接口只会增加契约面和维护成本。

现阶段最短闭环是复用现有轻链路，而不是扩接口。

### 11.3 为什么不直接延长 timeout

因为那只是把“35 秒超时”改成“60 秒超时”。

只要 full 继续承担错误职责，未来数据更重时还是会再回来。

## 12. 实施边界

这次实现只允许围绕以下模块展开：

- 客户端登录等待态与收口逻辑
- 轻量运行态到登录状态机的桥接
- full 补全态的页面状态表达
- 部署检查项的分类与补充

不在本次范围内的工作：

- 交易第三、四阶段其它功能回滚
- 重做服务端账户数据模型
- 重写 stream 协议

## 13. 未验证前提

以下前提当前是高概率成立，但在实施前仍需用测试锁定：

- 当前 `accountRuntime` / `snapshot` 的 identity 字段在登录刚建立时已经足够稳定
- 手机端 `timeout` 的最终触发点确实在 full 收口等待，而不是另有隐藏等待器
- full 慢的主要来源是完整账户快照构建本身，而不是 session/login 请求本身

这些前提在实现阶段必须通过针对性测试确认，不能只靠推测。

## 14. 结论

本次正式设计只做一件核心事：

把“账户登录成功”的定义，从“full 全量快照返回并匹配成功”，改成“轻量账户运行态确认目标账号已在线”。

这样可以把系统职责重新拉直：

- session/login 负责建立会话
- runtime/snapshot 负责确认会话是否已在线
- full 负责补完整数据

这才是和当前系统结构最一致、改动最小、闭环最短的正式修法。
