# 2026-04-17 架构性能配色清理设计

## 目标

在不改页面布局的前提下，一次性完成 7 类收口工作：

1. 收口旧入口和旧页面壳，避免同一页保留两套实现。
2. 找出图表与悬浮窗速度不一致的根因，并优先修掉最高频、最重的重复计算。
3. 提升 1 分钟图、图上历史交易、当前仓位/挂单叠加层的响应速度。
4. 统一全局配色入口，减少 XML、运行时 palette、图表内部硬编码互相打架。
5. 清掉前期调试和无业务价值的日志类型。
6. 记录当前 App 架构与网络传输架构的合理处、不合理处和下一步建议。

## 当前问题

- `MarketChartActivity` 已经退化为“永远桥接到主壳交易 Tab”，但文件里仍残留整套旧交易页实现，职责边界混乱。
- 图表叠加层的变化签名绑定了 `cache.updatedAt`，导致只要 stream 每秒推一次账户运行态，图表就容易把“无关更新”误判成“必须重建整套标注”。
- 历史成交锚点采用“每条成交线性扫描完整 candles 窗口”的做法，1 分钟窗口最容易放大成本。
- `AccountStatsPreloadManager.applyPublishedAccountRuntime()` 每次运行态更新都“写库 -> 读库 -> build cache”，I/O 往返落在高频链路上。
- 主题同时由 `colors.xml/themes.xml/styles.xml` 和 `UiPaletteManager.PALETTES` 两套入口维护，旧色值也散落在 drawable / 图表代码里。
- `ChainLatencyTracer`、`RemoteSessionDebug`、APP 诊断地址输出等日志默认开启，已经超出日常运行需要。

## 设计方案

### 1. 交易页旧架构收口

- 保留 `MarketChartActivity` 作为兼容桥接入口和路由常量承载层。
- 删除其不可达的旧页面实现，让真实交易页只保留在 `MarketChartFragment + MarketChartScreen`。

### 2. 图表性能主线

- 图表叠加层签名只保留“当前产品 + 当前可见窗口 + 真正影响标注的持仓/挂单/成交内容”，不再把 `updatedAt` 当作强制重建条件。
- 历史成交锚点改成基于已排序 candles 的二分定位，避免 1 分钟窗口反复做线性扫描。
- `applyPublishedAccountRuntime()` 直接用本次生成的运行态快照构建内存 cache，不再每秒读回持久层。

### 3. 配色统一

- 统一静态 token：背景、卡片、输入框、描边、文字、强调色重新定义为同一套深色终端风格。
- 更新 `UiPaletteManager` 的动态 palette，使运行时主题和 XML token 方向一致。
- 替换明显硬编码资源色：日志级别背景、旧 tab 背景、箭头图标、图表指标线颜色。

### 4. 日志清理

- 默认关闭 `ChainLatencyTracer`。
- 去掉 `MonitorService` 的 APP 诊断地址输出。
- 把远程会话 debug 和交易可见性 debug 从默认日志路径里移除。

## 预期结果

- 图表与悬浮窗的速度差主要收敛在“图表需要画图”这一层，而不是“每秒重复重建无关数据”。
- 1 分钟图下历史交易和当前仓位叠加层的重算频率、单次耗时都下降。
- 配色入口统一后，背景、按钮、下拉、抽屉、弹窗、边框、文字颜色不再各自漂移。
- 默认日志回到“业务结果 + 真错误”为主，调试噪音下降。
