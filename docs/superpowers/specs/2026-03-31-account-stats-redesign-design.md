# 账户统计重算与页面改版设计

## 1. 背景

当前账户统计页已经具备净值/结余曲线、收益表、交易统计与设置页分级入口，但还存在三类核心问题：

- 净值曲线口径过粗，和结余曲线过于接近，不能真实反映持仓浮动。
- 图表体系不完整，缺少回撤曲线、日收益率曲线、交易分布图、持仓时间分布图。
- 账户统计页和设置页仍保留较多圆角和卡片式容器，和用户要求的“更平、更直、更像专业终端 / 微信设置页”不一致。

用户已经确认本轮采用推荐方案：服务端重算净值曲线，客户端统一重画统计页和设置页。

## 2. 本次目标

本次实现同时覆盖以下 9 项需求：

1. 净值/结余曲线中的最大回撤区间高亮更明显，并且在任意所选周期下同步重算与标记。
2. 服务端基于历史成交、持仓、开平仓时间、产品价格、手数重算净值曲线，让净值与结余真正区分开。
3. 在净值/结余曲线下新增“回撤曲线”和“日收益率曲线”附图。
4. 月收益表改为参考图中的热力表风格，并统一日收益、年收益、阶段收益表的视觉语言。
5. 新增历史交易分布图，横轴最大回撤，纵轴收益率。
6. 新增持仓时间分布图。
7. 交易统计区的文字排版参考 `Key Performance Metrics`，但金额收益率保留红绿显示。
8. 页面与次级页面减少圆角，去卡片化。
9. 设置页改成更接近微信设置界面的列表式结构。

## 3. 已确认边界

- 服务端负责输出更可信的 `curvePoints`，客户端不再自行伪造“看起来不同”的净值。
- 仍保持当前 `/v1/curve` 接口为客户端主要曲线入口，不新增一套并行图表协议。
- 客户端新增图表优先复用现有账户数据、交易数据与曲线数据，不重复发额外大请求。
- 设置页保留现有功能项与分级结构，只改视觉骨架与交互呈现。
- 交易统计中的红绿显示仅保留在金额、收益率、盈亏相关数字上，其余文案改为中性专业排版。

## 4. 方案对比

### 方案 A：只改客户端展示

优点：

- 改动集中在 Android 端。

缺点：

- 净值曲线仍缺少真实历史浮盈路径。
- 容易继续出现净值与结余基本重合的问题。

### 方案 B：服务端重算曲线，但页面只做小改

优点：

- 能解决净值口径问题。

缺点：

- 图表体系与页面风格仍然不满足本轮目标。

### 方案 C：服务端重算曲线，客户端统一重画统计页和设置页（采用）

优点：

- 同时解决数据真实性与界面表达问题。
- 最大回撤、回撤曲线、日收益率、收益表、交易分布都能基于同一套数据口径协同工作。

缺点：

- 改动范围较大，需要桥接端和 App 端一起联动。

## 5. 采用方案

采用方案 C。

原因：

- 用户最核心的不满点不只是样式，而是“净值曲线不可信 + 最大回撤不明显 + 页面风格不够专业”。
- 这三件事拆开做会反复返工，合并处理才是最短路径。

## 6. 服务端曲线重算设计

### 6.1 新口径

服务端 `bridge/mt5_gateway/server_v2.py` 中的 `_build_curve` 不再简单把成交盈亏累计成一条近似结余线，而是改成：

- `balance`：按成交真实落袋盈亏递进。
- `equity`：在每个采样时点 = 当时结余 + 当时仍未平仓仓位的浮动盈亏。

### 6.2 重算输入

重算时使用以下信息：

- `history_deals_get` 返回的成交明细
- `positions_get` 返回的当前持仓
- 成交中的开平仓方向、开平仓时间、成交价、手数
- `symbol_info(...).trade_contract_size`
- MT5 历史价格序列（用 `copy_rates_range` 拉取不同周期下的 close）

### 6.3 重算方式

采用“事件点 + 适度采样点”的混合时间轴：

- 事件点：开仓、加仓、减仓、平仓、反手成交时间
- 采样点：按周期自动选择粒度补点，避免只有成交时间点导致净值过于锯齿或长期横线
- 结束点：当前时刻

每个时间点都重新计算：

- 当前结余
- 当前仍存活仓位
- 每个存活仓位按历史价格标记后的浮盈亏
- 汇总得到当时净值

### 6.4 退化策略

如果某个产品在历史价格拉取失败：

- 回退为仅用成交事件点与最新价计算
- 保证接口可返回，不因单个品种缺历史价而整段曲线失败

## 7. 客户端曲线与副图设计

### 7.1 主图

`EquityCurveView` 继续作为主图，但调整成更高对比的专业终端风：

- 净值线与结余线使用明显区分的双色
- 最大回撤区间使用高对比浅底色 + 深色边线
- 起点与谷底分别用不同形状或更大标记强调
- 回撤区间按当前显示区间实时重算，而不是只看全量

### 7.2 附图

在主图下增加两个附图控件：

1. 回撤曲线
   - 以主图当前显示区间为准
   - 纵轴为负向百分比
   - 与主图共享时间轴

2. 日收益率曲线
   - 基于当前区间内按日收盘净值计算
   - 用柱线或细柱图表达正负收益
   - 与主图共享时间轴

## 8. 收益表设计

### 8.1 月收益表

月收益表改为“年份 + 12 个月热力格”的结构：

- 正收益用绿阶
- 负收益用红阶
- 零或缺失用浅中性底色
- 年度汇总也采用统一网格风格

### 8.2 日 / 年 / 阶段收益表

统一遵循同一套语言：

- 表头简洁
- 单元格平直，不再强卡片感
- 点击某格后仍可联动主图区间

## 9. 新增交易统计图设计

### 9.1 历史交易分布图

新增散点图：

- 横轴：单笔交易生命周期中的最大回撤
- 纵轴：单笔交易最终收益率
- 点颜色：正负收益分色
- 特殊高收益或极端回撤点可加轻量强调

### 9.2 持仓时间分布图

新增分布柱图：

- 横轴：持仓时长分桶（如 0-1h、1-4h、4-12h、1d、3d、7d+）
- 纵轴：交易数量
- 用于快速看策略偏向短线还是持有型

## 10. 交易统计文字区设计

`recyclerStats` 的指标呈现从“短卡片块”改成更接近清单表：

- 左列指标名
- 右列数值
- 指标之间用细分隔线分组
- 金额、盈亏、收益率继续红绿
- 非盈亏指标保持中性色

## 11. 页面骨架设计

### 11.1 账户统计页

- 外层容器从大量 `MaterialCardView` 改为平直分段区块
- 减少圆角和重背景
- 更强调分隔线、标题、留白层次

### 11.2 设置页

保留“首页 -> 二级页”的结构，但样式改为微信式：

- 首页用整行列表项 + 分组间距 + 右箭头
- 二级页也用列表分段，而不是大卡片包小卡片
- 开关、输入框、按钮放进列表项语义中
- 主题选择区从大卡片改成更扁平的主题行或紧凑预览块

## 12. 并行实施拆分

本轮明确拆成两个并行子任务：

### 子任务 A：服务端曲线重算

负责：

- `bridge/mt5_gateway/server_v2.py`
- `bridge/mt5_gateway/tests/*`
- 曲线重算、价格采样、净值/结余分离

### 子任务 B：客户端页面与图表改版

再分成两块并行推进：

- 设置页微信式骨架
- 账户统计页主图、副图、收益表、统计图、去卡片化

## 14. 错误处理

- 服务端历史价缺失时必须降级，不让 `/v1/curve` 直接失败。
- 客户端如果拿到旧曲线或无副图数据时，至少保证主图和收益表可显示。
- 新增图表在无数据时显示明确空状态，不画残缺坐标。

## 15. 测试策略

至少覆盖以下验证：

1. 服务端曲线重算后，存在开仓未平的情况下，`equity` 与 `balance` 不再恒等。
2. 当前区间切换到 `1D / 7D / 1M / 3M / 1Y / ALL` 时，最大回撤区间都会重算。
3. 月收益表热力格能正确按正负收益着色。
4. 历史交易分布图与持仓时间分布图在有 / 无数据时都能稳定渲染。
5. 设置首页和二级页已不再依赖重卡片容器。

## 16. 非目标

- 不改当前底部导航结构
- 不新增新的账户接口页面
- 不把所有主题系统推倒重做
- 不修改交易记录筛选逻辑本身

## 17. 实施顺序

1. 先写设计文档和实施计划。
2. 服务端先补失败测试并重算曲线。
3. 设置页并行去卡片化，改成微信式列表。
4. 本地完成账户统计页主图、副图、收益表、交易统计图接入。
5. 最后统一验证、更新上下文文档。
