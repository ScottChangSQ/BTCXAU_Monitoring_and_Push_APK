# Windows 单窗口部署与连接状态面板设计

## 1. 背景

当前 Windows 部署链已经有一个前台可见的 `MT5 部署状态` 窗口，但服务器部署完成后，网关和管理面板仍会各自留下 PowerShell 常驻窗口。  
这会带来两个问题：

- 桌面上存在多个长期可见 PowerShell 窗口，无法保证“只有一个对人看的正式状态入口”
- 关键状态分散在部署窗口、后台黑窗、日志和接口里，排查时仍要来回切换

本次目标不是单纯“隐藏某个窗口”，而是把 Windows 服务器上的**可见状态真值**正式收口为一个长期存在的前台窗口，其余守护进程全部后台化。

## 2. 目标

本次设计要实现以下结果：

- 服务器桌面上只保留一个长期可见窗口，正式名称为 `MT5 部署与连接状态`
- 现有部署窗口在部署完成后不关闭，直接转为长期状态总面板
- 网关、管理面板等常驻 PowerShell 进程全部隐藏到后台运行
- 状态面板持续展示四类关键信息：
  - 服务器本地部署情况
  - 行情连接情况
  - MT5 账户与会话情况
  - 与手机 APP 的最近交互情况
- 状态展示只基于正式结构化状态，不依赖日志关键词猜测

## 3. 非目标

本次不做以下事情：

- 不把长期状态查看迁移到 Web 管理面板
- 不把部署链改造成 Windows Service
- 不新增第二个独立前台状态程序
- 不通过日志后处理推断“连接是否正常”
- 不改变现有网关、管理面板、Caddy 的核心运行职责

## 4. 当前现状

### 4.1 现有部署窗口

`deploy/tencent/windows/deploy_bundle.ps1` 已有单窗口 GUI，当前能力是：

- 前台显示部署步骤、部署日志、健康检查结果
- 后台隐藏启动一个部署 worker
- 部署结束后提示“可关闭此窗口”

当前缺口是：

- 这个窗口只覆盖“部署阶段”，没有正式进入长期运行期
- 没有单独的连接状态区
- 没有手机 APP 交互状态真值

### 4.2 现有后台守护链

`run_gateway.ps1` 与 `run_admin_panel.ps1` 当前都是无限循环守护脚本。  
它们负责在进程退出后重启服务，但在交互式用户上下文下运行时，会形成长期 PowerShell 可见窗口。

### 4.3 现有可复用状态真值

已有正式状态来源如下：

- `/health`
  - 网关健康、bundle 指纹、MT5 配置、最近 MT5 连接路径等
- `/v1/source`
  - 行情运行时状态，如 Binance WS 是否已连接、最近更新时间、最后错误
- `/v2/session/status`
  - 当前会话是否已激活、当前活动账号、已保存账号数量
- `/v2/session/diagnostic/latest`
  - 最近一次登录/切号链的阶段与失败信息

当前仍缺少一类正式真值：

- 手机 APP 最近交互状态
  - 当前是否有活跃 `v2/stream` 客户端
  - 最近一次客户端连接/断开时间
  - 最近一次会话请求与交易请求的结果

## 5. 设计结论

采用“**单前台窗口 + 全部后台守护 + 网关补结构化运行时状态**”方案。

### 5.1 总体结构

系统分成 3 层：

1. 后台服务层
- `run_gateway.ps1`
- `run_admin_panel.ps1`
- Caddy
- 计划任务

2. 状态真值层
- `deploy_bundle.ps1` 现有部署状态文件
- 网关现有 `/health`、`/v1/source`、`/v2/session/status`、`/v2/session/diagnostic/latest`
- 新增网关运行时状态接口，用于输出 APP 交互状态

3. 单窗口展示层
- 继续使用 `deploy_bundle.ps1` 的 WinForms 窗口
- 部署阶段显示部署过程
- 部署完成后切换为长期连接状态总面板

### 5.2 前台窗口职责

唯一前台窗口负责两件事：

- 部署期：显示部署步骤、日志、健康检查结果
- 运行期：持续显示当前连接状态总览

窗口本身不负责业务判断，只负责：

- 定时读取部署状态文件
- 定时访问本地状态接口
- 定时检查本地端口、任务和进程是否仍存活
- 把这些结构化状态渲染成统一界面

### 5.3 后台守护职责

所有守护 PowerShell 保持原有“退出即重启”的职责，但不再承担任何人机展示职责。

后台守护链只做：

- 保活
- 写日志
- 暴露正式状态

它们不再通过可见控制台窗口向用户传递状态。

## 6. 窗口信息架构

长期状态总面板分为 5 个区块。

### 6.1 顶部总览区

展示全局摘要：

- 当前总状态：正常 / 警告 / 异常
- 当前部署版本：bundleFingerprint
- 最近状态更新时间
- 部署目录
- 一键复制关键状态摘要按钮

### 6.2 本地部署区

展示本机部署链是否还活着：

- 网关端口 `8787`
- 管理面板端口 `8788`
- 反向代理 `80 / 443`
- 计划任务 `MT5GatewayAutoStart`、`MT5AdminPanelAutoStart`
- 网关守护脚本是否在运行
- 管理面板守护脚本是否在运行
- 最近部署结果与最近错误

### 6.3 行情连接区

展示市场链状态：

- 当前模式，如 `binance-ws`
- Binance WS 是否已连接
- 是否处于重连中
- 最近行情更新时间
- 最近行情错误

### 6.4 账户连接区

展示 MT5 与会话状态：

- MT5 是否已配置
- 当前是否已有 MT5 可用连接
- 最近连接到的终端路径
- 当前会话状态：`activated / logged_out`
- 当前活动账号与服务器
- 最近一次会话诊断阶段
- 最近一次失败原因

### 6.5 手机 APP 交互区

展示手机侧与服务器的真实交互状态：

- 当前活跃 `v2/stream` 客户端数
- 最近一次客户端连接时间
- 最近一次客户端断开时间
- 最近一次客户端普通请求时间
- 最近一次会话请求时间、动作、结果
- 最近一次交易请求时间、动作、结果
- 最近一次客户端来源标识

这里的“手机 APP 状态”定义为“客户端最近是否确实连过、请求过、成功过”，不是对用户意图做推断。

## 7. 状态来源设计

### 7.1 部署与本地存活状态

来源：

- `deploy_bundle.ps1` 当前已有的状态文件与日志文件
- 本地端口探测
- 计划任务查询
- 本地守护进程查询

这些状态继续由 PowerShell 侧负责，因为它们属于本机运行环境真值。

### 7.2 行情与账户状态

来源：

- `GET /health`
- `GET /v1/source`
- `GET /v2/session/status`
- `GET /v2/session/diagnostic/latest`

这些接口已经存在，本次不重新定义业务口径，只做消费。

### 7.3 手机 APP 交互状态

需要在 `bridge/mt5_gateway/server_v2.py` 新增一份正式运行时状态，通过以下接口暴露：

- `GET /internal/runtime/status`

该接口返回：

- `streamClientsActive`
- `streamLastConnectedAt`
- `streamLastDisconnectedAt`
- `httpLastRequestAt`
- `httpLastRequestPath`
- `sessionLastAction`
- `sessionLastRequestAt`
- `sessionLastResult`
- `tradeLastAction`
- `tradeLastRequestAt`
- `tradeLastResult`
- `lastClientAddress`

这份状态只表达“最近交互事实”，不做额外推理。

## 8. 网关运行时状态设计

### 8.1 统一运行时容器

在 `server_v2.py` 增加线程安全的运行时状态容器，专门记录：

- WebSocket 客户端连接数与最近连接事件
- 关键 HTTP 请求最近事实
- 最近会话动作结果
- 最近交易动作结果

### 8.2 写入时机

写入规则如下：

- `/v2/stream` 建连成功时：
  - 增加活跃客户端数
  - 写入最近连接时间
- `/v2/stream` 断开时：
  - 减少活跃客户端数
  - 写入最近断开时间
- `/v2/session/login|switch|logout`
  - 记录动作名、请求时间、成功或失败结果
- `/v2/trade/check|submit|batch/submit`
  - 记录动作名、请求时间、成功或失败结果
- 其他 `/v2/*` 请求
  - 记录最近普通请求时间和路径

### 8.3 读取方式

状态窗口只读取这个统一接口，不直接扫日志，不直接复原业务链。

## 9. PowerShell 与计划任务设计

### 9.1 后台化原则

除了单一状态窗口外，其他 PowerShell 都必须隐藏运行。

需要收口的入口包括：

- `deploy_bundle.ps1` 启动的后台 worker
- 计划任务启动的 `run_gateway.ps1`
- 计划任务启动的 `run_admin_panel.ps1`

### 9.2 计划任务启动行为

计划任务动作必须显式采用隐藏窗口方式启动 PowerShell，确保在交互用户登录后：

- 网关守护窗口不再出现在桌面
- 管理面板守护窗口不再出现在桌面

这里不改变计划任务的“交互用户优先，SYSTEM 兜底”策略，只改变其可见性。

### 9.3 状态窗口生命周期

部署窗口生命周期改为：

1. 用户双击部署脚本
2. 前台只出现一个窗口
3. 后台隐藏启动部署 worker
4. 部署成功后窗口不关闭
5. 窗口自动切换到长期状态模式
6. 用户关闭窗口时，只关闭展示层，不停止后台服务

## 10. 界面行为设计

### 10.1 部署阶段

保留现有内容：

- 步骤表
- 实时日志
- 健康检查结果

同时提前占位长期状态区，但在部署完成前显示为“等待部署完成”。

### 10.2 运行阶段

部署成功后：

- 顶部状态改为“运行中”
- 步骤区收起或保留为“最近部署结果”
- 重点切换到长期连接状态区
- 状态刷新节奏固定为每 1 秒一次

### 10.3 异常展示

任何一类状态异常时，窗口必须明确显示：

- 哪个模块异常
- 最近一次成功时间
- 最近错误文本

不能只显示模糊的“失败”或“未连接”。

## 11. 测试设计

需要补 3 组验证。

### 11.1 PowerShell 合同测试

覆盖：

- 计划任务脚本必须以隐藏方式启动守护 PowerShell
- 部署窗口脚本仍然只启动一个前台窗口
- 部署完成后窗口不会自动退出

### 11.2 网关合同测试

覆盖：

- `/internal/runtime/status` 返回固定结构
- `v2/stream` 建连和断开会正确更新客户端计数与时间
- 会话请求和交易请求会更新最近交互事实

### 11.3 集成行为验证

覆盖：

- 部署后桌面只有一个可见窗口
- 关闭状态窗口后，网关和管理面板继续运行
- 手机 APP 连接、断开、登录、交易后，状态面板能看到更新时间变化

## 12. 风险与约束

### 12.1 风险

- 如果只隐藏窗口但不补运行时真值，状态面板会退化成“日志查看器”
- 如果把所有状态都塞进 `/health`，会把健康检查接口和运行时观测职责混在一起

### 12.2 约束

- 状态面板必须消费正式结构化字段
- 后台守护链不能依赖前台窗口存在
- 前台窗口关闭不能影响后台服务

## 13. 最终结论

本次需求的最小完整方案是：

- 保留并升级现有 `MT5 部署状态` 窗口，变成唯一长期前台状态入口
- 网关、管理面板等守护 PowerShell 全部隐藏到后台
- 新增网关运行时状态接口，正式暴露手机 APP 交互事实
- 状态窗口统一汇总“部署、本地存活、行情、账户、手机交互”五类状态

这样可以把服务器端“人看的状态入口”和“后台运行入口”彻底分开，最终实现：  
**桌面只看一个窗口，后台只跑隐藏守护，状态只认结构化真值。**
