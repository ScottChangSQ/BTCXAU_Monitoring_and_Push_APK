# 统一市场真值中心设计

## 1. 背景

这轮排查已经确认，当前 APP 内部同时存在两条会直接影响显示的市场数据链：

1. `stream/仓库链`
   `MonitorService -> MonitorRepository -> MarketRuntimeSnapshot`
   这条链主要驱动悬浮窗和图表实时尾巴。
2. `REST/图表链`
   `GatewayV2Client.fetchMarketSeries* -> MarketChartSeriesLoaderHelper -> MarketChartScreen.applyDisplayCandles(...)`
   这条链主要驱动图表主显示序列。

当前问题不是某一个接口慢，而是这两条链没有在进入 UI 前合成为同一份真值：

- 悬浮窗最新价只吃 `MonitorRepository.selectLatestPrice(symbol)`。
- 图表主显示序列还能被页面自己的 REST 刷新继续推进。
- `5m~1d` 虽然有本地聚合代码，但它依赖页面内存里的 `minuteKey` 缓存，不是系统级正式底稿。

所以才会同时出现三个现象：

1. `1m` 图偶尔在动，但悬浮窗最新价长时间不动。
2. `5m~1d` 不跟随 `1m` 更新，也不会稳定重算。
3. 图表、悬浮窗、悬浮提示各自看到的“最新市场状态”不是同一份东西。

## 2. 目标

本轮设计目标只有一个：把 APP 的市场显示链收口成“一个正常的行情流 + 一个主显示流”。

具体要求：

1. APP 内部只保留一个正式市场真值出口，图表、悬浮窗、悬浮提示都只读这一个出口。
2. 最新价必须和 `1m` 实时 K 线使用同一份数据。
3. `5m~1d` 必须按 `1m` 底稿正式重算，不能再依赖页面局部缓存。
4. 历史断档必须能被系统明确发现，并按缺口范围正式补齐。
5. 实时更新不能依赖页面是否正在某个周期，也不能依赖用户手动切周期自愈。
6. 方案不能做成“每次来一跳就全量重算”，必须是增量合成，避免图表卡顿。

## 3. 非目标

- 本轮不改服务端协议定义。
- 本轮不在 UI 层继续补单点兜底。
- 本轮不让页面继续自己维护一套正式 `minuteKey` 真值。
- 本轮不引入“悬浮窗一套、图表一套”的显示兼容分支。

## 4. 方案对比

### 方案 A：`stream` 单独做真值

优点：

- 链短，实时性最好。

缺点：

- 历史断档、补窗、切周期窗口都不够稳定。
- 一旦 `stream` 某段时间没推进，整套显示都会卡住。
- 不能完整承担正式历史底稿职责。

### 方案 B：`REST` 单独做真值

优点：

- 历史窗口完整性最好。

缺点：

- 最新价和当前未闭合 K 线实时性不够。
- 悬浮窗和图表都会变成“轮询刷新感”，不像正常行情流。

### 方案 C：`stream + REST` 先合成，再统一对外（采用）

优点：

- `stream` 负责推进当前正在走的市场状态。
- `REST` 负责补正式历史、补缺口、补切周期窗口。
- UI 最终只看到一条正式读链。

缺点：

- 需要新增一个统一市场中心，把现在分散在页面和仓库里的职责重新收口。

## 5. 采用方案

采用方案 C，但要进一步收紧成以下原则：

1. `1m` 是唯一正式基础时间粒度。
2. 最新价、`1m` 实时尾巴、`5m~1d` 最新未闭合条都从同一份 `1m` 实时状态推导。
3. `5m~1d` 的正式闭合历史由系统级投影层从 `1m` 闭合底稿增量生成，不再由页面本地临时聚合。
4. `REST` 进入系统后不再直接渲染页面，只能写入统一市场中心。
5. 页面只消费统一市场中心发布的读模型。

## 6. 核心架构

### 6.1 MarketTruthCenter

这是新的唯一市场真值入口，负责接住所有市场输入，并产出唯一正式输出。

它只做五件事：

1. 接收 `stream` 实时快照。
2. 接收 `REST` 历史窗口和修补结果。
3. 维护 `1m` 正式底稿与当前未闭合分钟草稿。
4. 增量投影出 `5m~1d`。
5. 对 UI 发布统一读模型。

`MonitorRepository` 后续不再直接扮演“悬浮窗最新价真值 + 图表实时尾巴真值”的最终角色，而是退成统一市场中心的一个数据出口或兼容壳。

### 6.2 MinuteBaseStore

这是唯一正式的 `1m` 基础库，按 `symbol + openTime` 存储。

它内部区分两类状态：

1. `closed minute`
   已闭合、已确认的正式 `1m` K 线。
2. `draft minute`
   当前正在走的未闭合分钟，随着 `stream` 实时推进。

显示口径固定为：

- 悬浮窗最新价 = `draft minute` 的最新成交价。
- 图表 `1m` 最新一根 = `closed minutes + draft minute`。
- 高周期最新一根 = `closed minutes` 投影结果 + 当前 `draft minute` 折算出来的实时尾巴。

### 6.3 IntervalProjectionStore

这是系统级高周期投影层，负责维护 `5m / 15m / 30m / 1h / 4h / 1d` 的正式闭合桶。

它不是页面缓存，而是统一市场中心的一部分。  
它只接受 `MinuteBaseStore` 的增量变化，不直接接受页面请求。

规则固定为：

1. 每当有一个 `1m closed minute` 落库，只重算它所属的那几个高周期桶。
2. 只更新受影响的最后一个桶，不做全量重刷。
3. 当前未闭合高周期条不落成正式历史，只在读模型阶段用 `draft minute` 做实时尾巴投影。

### 6.4 GapDetector

这是断档检测器，负责判断当前正式底稿是否连续。

它不靠“看起来像旧了”这种经验判断，而是按标准时间步长做确定性检查：

1. `1m` 检查相邻 `openTime` 是否严格相差 60 秒。
2. 高周期检查相邻桶是否严格相差对应周期步长。
3. 当前分钟推进时，若发现上一闭合分钟缺失，立即标记缺口。
4. `REST` 拉回一段历史后，再次检查该段首尾是否连续。

### 6.5 HistoryRepairCoordinator

这是正式补档调度器，负责把缺口修平。

它只做两种补法：

1. 小范围缺口：按缺口时间窗定点补拉 `1m` 历史。
2. 大范围不可信：重拉当前活动窗口对应的 `1m` 正式区间，再重新贴回实时尾巴。

补档永远以 `1m` 为主，不再让页面直接拉某个周期后自己当正式结果显示。

### 6.6 UiReadModelAdapter

这是统一对 UI 的只读出口。

它至少提供四类结果：

1. `latest price snapshot`
2. `1m display series`
3. `5m~1d display series`
4. `hover / crosshair / floating snapshot`

这些结果都从同一套底稿读取，不能再分别从仓库链和页面链拿。

## 7. 具体数据流

### 7.1 实时推进流

1. `stream` 到达。
2. `MarketTruthCenter` 先更新当前 `draft minute`。
3. 立刻发布新的 `latest price snapshot`。
4. 若当前选中的是 `1m`，直接把 `draft minute` 拼到 `1m display series` 尾部。
5. 若当前选中的是 `5m~1d`，只重建“当前最后一根未闭合高周期条”，不重建整个序列。
6. 悬浮窗、图表、悬浮提示同时收到同一轮新快照。

这里的关键是：每次实时跳动只触碰“当前分钟草稿”和“各周期最后一根显示尾巴”，不会触碰整段历史。

### 7.2 分钟闭合流

1. 当系统确认某个 `1m` 已闭合后，把该分钟从 `draft` 转成 `closed minute`。
2. `MinuteBaseStore` 落正式底稿。
3. `IntervalProjectionStore` 只更新这个分钟所属的 `5m~1d` 桶。
4. UI 收到新的闭合投影结果。

这一步的工作量是固定的：

- 一个 `1m` 闭合，只会影响有限几个高周期桶。
- 不会因为一根分钟闭合就回头重刷整张图。

### 7.3 冷启动 / 切产品 / 切周期流

1. UI 不再直接请求“把结果塞进页面缓存”。
2. 页面只告诉 `MarketTruthCenter`：当前需要哪个产品、哪个周期、哪个时间窗。
3. 若本地真值缺该段 `1m` 底稿，由 `HistoryRepairCoordinator` 发起正式补拉。
4. 拉回结果先写 `MinuteBaseStore`，再由投影层产出对应周期序列。
5. 页面只接收统一市场中心给出的显示序列。

所以切周期时，页面不再自己临时聚合，也不再自己决定哪份历史可信。

## 8. 如何做到“不会卡”

这个方案不卡的关键，不是“本地少算一点”，而是“严格限制每次只算被影响的最小范围”。

### 8.1 实时阶段只动尾巴

实时跳动时：

- `1m` 只更新当前 `draft minute`
- `5m~1d` 只更新当前最后一根显示尾巴

所以每次 tick 的成本是常数级，不随整张图长度变大。

### 8.2 闭合阶段只动受影响桶

每个 `1m` 闭合时：

- 只更新这 1 根分钟所属的高周期桶
- 不是重新把全量 `1m` 再聚合一遍

因此 `5m~1d` 的正式重算是“增量投影”，不是“全量再算”。

### 8.3 历史补档放后台

补档、补窗口、修复断档统一在后台线程做：

- 先补 `MinuteBaseStore`
- 再局部刷新受影响窗口
- 最后把不可变结果快照发给 UI

主线程只做渲染，不做历史聚合和缺口修复。

### 8.4 读写分离

写侧负责：

- 吞 `stream`
- 接 `REST`
- 维护底稿
- 维护投影

读侧负责：

- 给图表一段 ready-to-render 的序列
- 给悬浮窗一个 ready-to-show 的最新快照

这样周期切换只是换读模型，不会触发页面现场重算。

## 9. 断档如何知道、如何补

### 9.1 如何知道

系统不再用“最后一根看起来还新鲜”来判断，而改成三类确定性信号：

1. 时间连续性信号  
   相邻 `1m` 是否严格连续。
2. 边界推进信号  
   当前分钟已经进入更晚时刻，但上一闭合分钟仍缺失。
3. 活动窗口完整性信号  
   当前图表所需时间窗内，是否存在缺桶。

只要任一信号不成立，该段就进入 `gap pending repair` 状态。

### 9.2 如何补

补档顺序固定为：

1. 先补 `1m` 缺口。
2. `1m` 补齐后，自动重建受影响的高周期桶。
3. 最后重新生成当前周期显示序列。

这意味着高周期永远不单独“猜着补”，而是依赖 `1m` 正式底稿恢复一致性。

### 9.3 大缺口处理

如果当前时间窗内缺口过大，或者状态已经不可信：

1. 直接重拉当前活动窗口的整段 `1m` 正式历史。
2. 用返回结果覆盖该窗口对应的 `MinuteBaseStore` 分区。
3. 重新投影出 `5m~1d`。
4. 再把当前 `draft minute` 贴到最新尾巴。

这样修的是“正式底稿”，不是在显示层打补丁。

## 10. 与当前运行方案的区别

### 10.1 当前方案

- `stream` 和 `REST` 都能直接影响最终显示。
- 悬浮窗主要吃仓库链。
- 图表主序列主要吃页面 REST 链。
- `5m~1d` 实时重算依赖页面内存 `minuteKey`。
- 页面自己承担了部分正式真值职责。

### 10.2 新方案

- `stream` 和 `REST` 都只能先进入统一市场中心。
- UI 不再直接消费任一输入链。
- `1m` 成为唯一正式基础底稿。
- `5m~1d` 由系统级投影层正式生成。
- 悬浮窗、图表、悬浮提示只读统一读模型。

一句话概括：

当前是“两条输入链分别改显示”，新方案是“两条输入链先合成，再统一出显示”。

## 11. 模块迁移边界

### 11.1 需要下沉出页面的职责

从 `MarketChartScreen` / `MarketChartDataCoordinator` 下沉出去：

- 页面内 `minuteKey` 正式底稿职责
- 高周期正式聚合职责
- 页面自己判断数据是否新鲜的职责
- 页面自己决定何时用 REST 修窗口的职责

### 11.2 需要保留在页面的职责

页面仍只负责：

- 当前产品/周期/视口选择
- 图表渲染
- 十字光标、交易叠加、视口跟随

页面不再拥有市场真值本身。

## 12. 测试要求

至少覆盖以下验证：

1. `latest price`、`1m` 最新尾巴、悬浮窗最新价三者始终同源。
2. 当 `stream` 推进但 `REST` 尚未刷新时，悬浮窗和 `1m` 同步变化。
3. 当 `1m` 闭合后，`5m~1d` 只更新受影响桶，且结果与 `1m` 一致。
4. 当前窗口出现分钟缺口时，系统能进入补档状态并在补完后恢复连续。
5. 切周期时不会触发页面级全量重算。
6. 自动刷新、冷启动、切产品、切周期四种入口都只经过统一市场中心。
7. 数据新鲜度判断不再依赖未闭合 K 线未来 `closeTime` 造成的假新鲜。

## 13. 实施顺序建议

1. 先建立 `MarketTruthCenter` 与 `MinuteBaseStore`，把 `stream` 和 `REST` 输入都收进来。
2. 再建立 `IntervalProjectionStore`，把 `5m~1d` 从页面聚合迁到系统级投影。
3. 再接 `GapDetector + HistoryRepairCoordinator`。
4. 最后把悬浮窗、图表、悬浮提示统一切到 `UiReadModelAdapter`。

这样可以先统一真值，再迁显示，不会一边迁 UI 一边继续扩散数据链。
