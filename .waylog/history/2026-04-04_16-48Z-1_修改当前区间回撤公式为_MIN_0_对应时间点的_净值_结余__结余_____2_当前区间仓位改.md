- 批量开仓
- 修改 TP/SL
- 图表拖拽改价后的确认提交

现在项目只有“读取账户状态”，没有“交易指令状态机”。实盘后必须新增：
- `Draft`：用户编辑中
- `Checking`：服务器预检查
- `Confirming`：等待用户确认
- `Submitting`：已发给服务器
- `Accepted / Rejected / Timeout`
- `Settled`：账户快照已同步回来

否则就会出现一个很危险的问题：用户以为已经下单，实际上只是点了按钮，但服务器拒单、超时、滑点重报价，你的页面却没有把这几种情况严格区分。

**2. 服务端必须从“读网关”升级成“交易网关”**
这不是 APP 先改 UI 能解决的，必须先改服务端。

服务端要新增的最小能力：
- `order_check`
- `order_send`
- 交易参数合法性校验
- 交易结果标准化返回
- 去重请求号
- 幂等处理
- 超时和重试策略
- 账户模式识别：`netting / hedging`
- 订单生命周期事件流

原因很直接：
- MT5 实盘不是“发一个请求就算成功”，还会遇到最小手数、步进、冻结距离、止损止盈距离、交易时间、品种停牌、保证金不足、重报价、滑点过大这些拒单情况。
- 这些判断不能放在手机端猜，必须由服务器统一校验，再把结果明确回给 APP。

建议服务端把交易接口拆成 3 层：
- `trade/check`
- `trade/submit`
- `trade/result or trade/events`

这样手机端才能先预演，再确认，再执行。

**3. 账户同步机制要从“轮询快照”升级成“交易后强一致刷新”**
现在项目很多地方还是“刷新一下快照看看结果”。展示没问题，实盘不够。

实盘要补的机制：
- 发单后立即进入“待确认”态
- 服务端返回受理结果后，立即触发该账户的增量刷新
- 直到最新持仓、挂单、成交、保证金都同步回来，页面才结束这次交易流程

否则会出现：
- 已经下单成功，但页面还是旧仓位
- 已经平仓成功，但图表还挂着旧止损线
- 已经改单成功，但订单列表里还是旧价格

这类问题在实盘里会直接损害信任。

**4. 图表能力要补成“可交易图表”**
你现在的图表页已经有很好的显示基础，但距离交易图表还差一整层“交互语义”。

需要新增：
- 持仓线、挂单线、SL 线、TP 线的独立模型
- 每条线的可选中状态
- 拖拽改价
- 拖拽后浮层显示新价格、点数、预估盈亏
- 图上直接平仓/删单/改单
- 图上直接新增挂单
- 图上直接添加或删除 TP/SL
- 图上区分“未确认草稿线”和“已生效订单线”

关键点不在画线，而在状态分层：
- 灰色：草稿
- 蓝色：已提交待确认
- 绿色/红色：已生效
- 橙色：服务器拒绝后回退提示

如果没有这层状态分离，图表交易会非常容易误导用户。

**5. 风控和确认流程必须单独设计**
这是实盘和展示型 APP 的分水岭。

至少要支持两套模式：
- 安全模式：每次开仓、平仓、改单都二次确认
- 快速模式：一键交易，但只对用户明确开启的功能生效

建议最少做这些控制：
- 默认关闭一键开仓/平仓
- 默认开启确认弹窗
- 大手数二次确认
- 批量操作强制确认
- 超过某个风险阈值时强制确认
- 显示预估保证金占用、点值、止损金额、止盈金额
- 显示净值变化和可用保证金变化

MT5 官方的一键交易之所以能做，是因为它有明确的开关和交互边界。你这边如果直接把“一键开仓/平仓”放出来，风险会很高。

**6. 批量交易能力不能直接堆在现有页面上**
“批量开仓 / 批量平仓 / 批量改 SLTP” 不能只是循环调接口。

必须补：
- 批次 ID
- 批次执行计划
- 成功/失败分项结果
- 中途终止策略
- 失败后的处理规则

例如“批量平仓 10 单”至少有 3 种模式：
- 尽力执行：能平多少平多少
- 全有或全无：有一单不满足就整批不发
- 分组执行：按品种或方向分组

这块如果不先定规则，后面很容易做成“看起来能用，出错后完全说不清”。

**7. 本地数据模型要补齐交易领域对象**
当前项目的模型更偏展示。实盘要新增一组“交易领域模型”。

至少需要：
- `TradeIntent`
- `TradeCheckResult`
- `TradeCommand`
- `TradeReceipt`
- `OrderDraft`
- `PositionAction`
- `BatchTradePlan`
- `ExecutionError`
- `BrokerConstraint`

因为“展示持仓”和“提交交易命令”不是一类对象。
现在 `PositionItem` 这类模型适合展示，不适合直接拿去发单。

**8. 安全、权限、审计要单独补**
如果接实盘，这一块不能省。

需要补的最低项：
- 交易密码或二次验证机制
- 一键交易单独授权
- 本地敏感配置加密
- 服务端令牌隔离
- 交易日志审计
- 每次指令的 requestId / accountId / symbol / volume / time / result 留痕
- 关键失败告警

尤其你现在是 Android + 自建 Windows 网关，这种结构一旦接实盘，安全问题比功能问题更先爆。

**9. 测试体系要升级**
现在的测试更偏页面逻辑和数据展示。实盘必须补这几类测试：

- 交易前校验测试
- MT5 返回码映射测试
- 滑点/拒单/重报价测试
- 网络断开重试测试
- 幂等重复提交测试
- 批量交易部分成功测试
- 图表拖拽改单测试
- netting / hedging 两种账户模式测试
- 真机压力测试

特别要注意 `netting` 和 `hedging`：
- 这会直接影响“加仓”“反手”“部分平仓”“close by”“图表显示几条线”
- 如果这层没先抽象清楚，后面很多功能会越做越乱

**如果按优先级排序，我建议这样分阶段**
最短正确路径不是一次做全，而是四阶段。

1. 第一阶段：打通最小交易闭环
- 市价开仓
- 单笔平仓
- 挂单新增/删除
- 修改单笔 TP/SL
- 统一确认弹窗
- 交易成功后强一致刷新

2. 第二阶段：做成可用的交易终端
- 图表订单展示
- 图表上改单/平仓
- 一键交易开关
- 订单详情和拒单原因
- 手数、保证金、盈亏预演

3. 第三阶段：补批量和高级能力
- 批量平仓
- 批量开仓
- 批量改 SL/TP
- close by
- netting / hedging 完整兼容

4. 第四阶段：补专业级体验
- DOM 交易
- 高级风险控制
- 快捷模板
- 常用下单参数记忆
- 交易日志与回溯

**综合判断**
如果你的目标是“尽量接近 MT5 官方移动端的实盘能力”，最需要优化的不是页面，而是这 4 个底层点：

- 交易命令链
- 服务端交易网关
- 交易后强一致同步
- 风控确认体系
